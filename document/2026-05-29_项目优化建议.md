# 水泥配比计算器项目优化建议

评估日期：2026-05-29

## 1. 总体判断

项目当前功能可用，后端 API 与前端主流程都能正常运行；黑盒测试范围内未出现阻断性故障。

因此，现阶段最值得投入的方向不是“补救已崩坏功能”，而是提升工程稳健性，降低后续维护和扩展成本。

建议优先级分为：

- P0：必须尽快处理，影响安全基线或生产风险
- P1：应尽快处理，影响维护成本、扩展速度和稳定性
- P2：建议处理中，影响性能、可读性和长期质量

## 2. P0 级建议

### 2.1 去除硬编码默认管理员密码

现状：`backend/database.py` 中存在固定默认管理员账号和密码：`admin / 123456`。

风险：

- 默认口令属于直接安全风险
- 代码仓库、打包文件或部署脚本泄露后，攻击门槛极低
- 当前黑盒测试也验证了该默认账号在本地环境确实可登录

建议：

- 用环境变量或首次初始化脚本注入管理员账号
- 首次启动仅在数据库为空时创建管理员
- 禁止把默认密码常量写进代码库
- 保留强制改密，但改为一次性初始化策略，而不是长期依赖默认口令

### 2.2 增加登录限流与审计日志

现状：`backend/routers/auth.py` 仅做用户名密码校验，未见登录限流、失败计数、锁定策略、IP 级审计。

风险：

- 抗暴力破解能力不足
- 出现异常登录时缺乏追踪证据

建议：

- 为登录接口增加用户名/IP 维度的限流
- 记录登录成功、失败、登出、改密、管理员重置密码等审计事件
- 把审计日志输出到独立日志文件或日志平台

### 2.3 不要让前端长期信任 `localStorage` 中的权限状态

现状：`frontend/src/utils/request.ts`、`frontend/src/stores/authStore.ts`、`frontend/src/router/index.ts` 都依赖 `localStorage` 中的 token 和 `sc_admin`。

风险：

- `localStorage` 容易受 XSS 影响
- `sc_admin` 可被客户端手工篡改，虽然服务端仍会拒绝越权接口，但会造成前端权限 UI 与真实权限不一致

建议：

- 优先改为 HttpOnly + SameSite Cookie 会话方案
- 如果短期内保持 Bearer Token，至少不要把管理员判断只交给本地缓存
- 页面初始化时通过 `/api/auth/profile` 回填真实权限状态

## 3. P1 级建议

### 3.1 拆分 `backend/database.py`

现状：`backend/database.py` 当前约 644 行，混合了以下职责：

- 数据库初始化与迁移
- 用户与密码逻辑
- 会话逻辑
- 项目 CRUD
- 记录 CRUD
- 权限判定

问题：

- 职责过多，修改时容易引发连带风险
- 难以做针对性测试与代码审阅
- 扩展新业务时会继续膨胀

建议拆分为：

- `backend/core/db.py`：连接、初始化、迁移
- `backend/repositories/users.py`
- `backend/repositories/sessions.py`
- `backend/repositories/projects.py`
- `backend/repositories/records.py`

这样后续无论接入 ORM、切换数据库，还是补集成测试，代价都会明显降低。

### 3.2 继续拆薄 HPC 前端主功能

现状：HPC 相关前端虽然已经做过一次拆分，但核心文件仍偏大：

- `frontend/src/views/HpcView.vue`：约 444 行
- `frontend/src/components/hpc-trial/HpcTrialStrengthTab.vue`：约 462 行
- `frontend/src/composables/useHpcTrial.ts`：约 677 行
- `frontend/src/stores/calcStore.ts`：约 328 行

问题：

- 视图编排、状态恢复、请求拼装、表格映射、公式展示仍然耦合较重
- 单文件理解成本高，后续改动容易牵动整块逻辑

建议：

- 让 `HpcView.vue` 只负责路由与页面装配
- 将 `useHpcTrial.ts` 再拆成“输入状态”“服务端同步”“快照恢复”“结果映射”几部分
- 将 `HpcTrialStrengthTab.vue` 再拆成“步长输入区”“强度输入区”“关系表格区”“回归推荐区”
- 为试配三个阶段建立更明确的 view-model 或 composable 边界

### 3.3 引入统一工程规范链

现状：

- `frontend/package.json` 只有 `dev/build/preview`，没有 `lint/test/typecheck` 命令
- 仓库中未发现 ESLint、Prettier、Ruff、Mypy、Pytest 等配置文件
- 前端目录未发现自动化测试文件

问题：

- 代码风格和导入顺序难以统一
- 小问题只能靠人工 review 发现
- 回归保护偏弱

建议：

- 前端补 `eslint + prettier + vue-tsc`
- 后端补 `ruff + pytest`，必要时加 `mypy`
- 增加 `pre-commit` 钩子
- 在 CI 中强制执行构建、静态检查和最小测试集

### 3.4 统一错误处理和日志策略

现状：

- `backend/main.py` 有兜底异常处理，但未见结构化日志输出
- 多个 router 中使用 `except Exception as e` 后直接拼接 `str(e)` 返回给客户端

问题：

- 客户端错误文案不稳定
- 内部异常信息可能被直接暴露
- 线上排障时缺少请求 ID、堆栈、用户上下文

建议：

- 定义统一错误码和错误响应模型
- 服务端记录完整异常日志，客户端只返回可控文案
- 加入请求 ID、中间件日志、慢请求日志

## 4. P2 级建议

### 4.1 优化前端包体

现状：本次 `npm run build` 结果中，主 JS 产物约为 1.58 MB，gzip 后约 488.72 kB。

可能原因：

- `frontend/src/router/index.ts` 全量静态导入页面组件，未做路由级懒加载
- `frontend/src/main.ts` 全量注册了所有 Element Plus 图标

建议：

- 采用路由级动态导入
- 仅按需注册实际使用的图标
- 对试配图表、公式说明等体积较大的功能做延迟加载

### 4.2 将配置外置

现状：

- `backend/main.py` 中的 CORS 白名单为硬编码
- `backend/database.py` 中的管理员初始化策略和会话 TTL 也写在代码中

建议：

- 增加 `backend/core/config.py`
- 使用环境变量管理 CORS、数据库路径、会话 TTL、管理员初始化参数
- 区分开发、测试、生产三套配置

### 4.3 完善注释策略，重点写“领域说明”而不是“翻译代码”

现状：

- 前端 store 内部已经有不少字段注释，这是优点
- 但后端公式、试配阶段转换、关键业务约束的说明仍不够集中

建议：

- 为核心计算服务补“输入含义、约束条件、公式来源、结果单位”说明
- 为 `/api/hpc-trial` 的请求和响应增加示例文档
- 为项目、记录、试配快照建立一份领域说明文档，降低新开发者上手成本

### 4.4 补齐测试金字塔

现状：

- 后端已有 2 个回归测试文件，覆盖面仍较有限
- 前端未见自动化测试

建议：

- 后端增加 auth/projects/records/upload-fit 的接口级集成测试
- 前端增加核心页面的组件测试
- 保留当前黑盒脚本，继续补 Playwright E2E 冒烟测试
- 把本次新增的 `document/run_blackbox_tests.py` 纳入回归验证流程

### 4.5 修正文档与实现不一致项

现状：根目录 `README.md` 中提到状态统一存储在 `src/stores/calcStore.js`，但当前实际文件为 `src/stores/calcStore.ts`。

建议：

- 更新 README 中的文件路径、技术栈说明、测试命令
- 把黑盒测试命令与构建命令一并写入 README

## 5. 建议的实施顺序

### 第一阶段：1 至 2 天

- 去除默认管理员密码硬编码
- 增加配置模块，外置 CORS 和安全相关参数
- 增加前后端 lint/typecheck 命令
- 更新 README

### 第二阶段：3 至 5 天

- 增加登录限流、审计日志、统一错误码
- 补 auth/projects/records/upload-fit 集成测试
- 前端接入基础 Playwright 冒烟脚本

### 第三阶段：1 至 2 周

- 拆分 `backend/database.py`
- 拆分 `useHpcTrial.ts`、`HpcView.vue`、`HpcTrialStrengthTab.vue`
- 推动路由懒加载和图标按需注册

## 6. 已完成的小修复

本轮工作中已顺手修复一个现存质量问题：

- 修复 `backend/database.py` 中 records 行转换辅助函数的静态类型声明问题
- 验证结果：Pylance 报错消失，后端现有单元测试继续通过

## 7. 结论

项目现在更像“可交付的功能原型”，而不是“已经完成工程化收口的产品版本”。

如果接下来要继续迭代，建议优先补齐安全和工程基线，再做结构拆分和性能优化。这样能避免后面功能越做越多、维护成本快速失控。
