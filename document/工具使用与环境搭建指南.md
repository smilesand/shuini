## 1. 项目总览

一套前后端业务代码，打包出 4 类产物：

| 产物 | 说明 | 是否加密数据库 | 是否需要激活 |
| --- | --- | --- | --- |
| Web 版 | 后端二进制 + 前端静态页，浏览器访问 | 否 | 否 |
| 桌面版 | Electron 外壳内置后端，单机运行 | **是（整库 AES-256-GCM）** | **是（RSA 授权码 / 3 天试用）** |
| 授权签发端 | 管理员给设备指纹签发授权码（内嵌私钥） | — | 仅内部使用 |
| 开发者工具 | 数据库解密/加密工具 + 试用期修改工具 | — | 可给客户自助验证 |

技术栈：

- 后端：Python + FastAPI + SQLite（运行期整库驻留内存，磁盘上始终是密文）。
- 前端：Vue 3 + Vite + Pinia + Vue Router + Element Plus。
- 桌面壳：Electron + electron-builder（NSIS 安装包）。
- 打包：`scripts/package.py`（统一入口）。

目录结构（关键部分）：

```
shuini_calculator/
├── backend/                # FastAPI 后端 + 业务/数据库逻辑
│   ├── core/secure_db.py   #   整库加解密 + license_state 防篡改
│   ├── core/_db_secret.py  #   内嵌数据库主密钥（由 init_license_keys 生成，已混淆）
│   ├── core/_license_pubkey.py # 内嵌授权公钥（已混淆）
│   └── database.py         #   schema / 迁移 / 旧库平滑升级
├── frontend/               # Vue 3 前端
├── desktop/                # Electron 外壳（main.cjs）
├── license_admin/          # 管理员授权签发端 GUI
├── scripts/
│   ├── package.py          # 统一打包脚本
│   ├── init_license_keys.py# 一次性生成全部密钥与内嵌模块
│   ├── decrypt_db.py       # 数据库解密/加密/查看工具
│   ├── expire_trial.py     # 试用期修改工具（验证到期行为）
│   ├── verify_legacy_upgrade.py # 旧版数据库平滑升级验证脚本
│   └── keys/               # 密钥源（private/public/db_key，需离线备份）
└── release/                # 各类打包产物输出目录
```

## 2. 一台新机器如何从零搭建开发/编译环境

> 以下为 **Windows 11** 步骤；Debian/Linux 见每步末尾备注。本项目当前在 **Windows + Debian** 原生打包，不做跨平台交叉编译。

### 2.1 必备运行时（已在本机验证的版本）

| 组件 | 版本 | 用途 |
| --- | --- | --- |
| Python | **3.14.5**（≥3.11 即可） | 后端、打包、全部脚本 |
| Node.js | **24.15.0**（≥18 即可） | 前端构建、Electron 打包 |
| npm | 11.12.1 | 随 Node 自带 |
| Git | 任意近期版本 | 拉取源码 |

安装建议：

- Python：从 python.org 安装，勾选 “Add python.exe to PATH”。
- Node：从 nodejs.org 安装 LTS 或上述版本。
- 验证：
  ```powershell
  python --version
  node --version
  npm --version
  ```

> Debian：`sudo apt install python3 python3-pip python3-venv` + 用 nvm 安装 Node。

### 2.2 拉取源码

```powershell
git clone <仓库地址> shuini_calculator
cd shuini_calculator
```

### 2.3 安装后端依赖

```powershell
cd backend
python -m pip install -r requirements.txt
python -m pip install -r requirements-dev.txt   # 含 PyInstaller、pytest 等，打包/测试需要
cd ..
```

后端运行期依赖（`backend/requirements.txt`）：

```
fastapi==0.115.0
uvicorn[standard]==0.30.6
numpy==1.26.4
scipy==1.13.1
python-multipart==0.0.9
openpyxl==3.1.5
cryptography==43.0.1
```

### 2.4 安装前端依赖

```powershell
cd frontend
npm install
cd ..
```

### 2.5 安装桌面壳依赖（仅打包桌面版需要）

```powershell
cd desktop
npm install
cd ..
```

### 2.6 生成密钥与内嵌模块（关键一步）

全新克隆出来的仓库，如果 `scripts/keys/` 或 `backend/core/_db_secret.py`、`backend/core/_license_pubkey.py` 缺失，**后端将无法加密数据库、无法校验授权码、打包会报错**。

- **要继续沿用现有客户已激活的授权码 / 已加密的数据库** → 必须从安全备份里 **还原原始的 `scripts/keys/` 三个文件**（见[第 7 节](#7-密钥体系与备份重要)），再运行：
  ```powershell
  python scripts/init_license_keys.py        # 已存在密钥则跳过生成，仅刷新内嵌模块
  ```
- **全新项目 / 允许作废历史授权码** → 直接生成全新密钥：
  ```powershell
  python scripts/init_license_keys.py --force
  ```

> ⚠️ `--force` 会重建 RSA 密钥与数据库主密钥：历史签发的授权码全部失效，旧的加密数据库将无法再被新主密钥解密。生产环境务必先确认。

### 2.7 验证环境可用

```powershell
# 后端单元测试
cd backend
python -m pytest -q
cd ..

# 前端构建自检
cd frontend
npm run typecheck
npm run build:web
cd ..

# 旧库平滑升级验证（见第 6 节）
python scripts/verify_legacy_upgrade.py
```

全部通过即代表环境搭建成功，可以开发与打包。

## 3. 日常开发与运行

### 3.1 启动后端（开发模式）

```powershell
cd backend
uvicorn main:app --reload --port 8000
```

- API 文档：http://localhost:8000/docs
- 默认 `SC_EDITION=web`（不加密、不强制激活），便于本地开发。
- 若要本地复现 **桌面版加密 + 授权** 行为，设置环境变量后再启动：
  ```powershell
  $env:SC_EDITION = "desktop"
  $env:SC_DB_PATH = "$PWD\data.db"
  python .\main.py
  ```

### 3.2 启动前端（开发模式）

```powershell
cd frontend
npm run dev          # http://localhost:5173，自动代理 /api 到 8000
```

### 3.3 关键环境变量

| 变量 | 默认 | 含义 |
| --- | --- | --- |
| `SC_EDITION` | `web` | `web`=不加密不激活；`desktop`=整库加密 + 授权强制 |
| `SC_DB_PATH` | 见 config | 数据库文件路径（桌面版由 Electron 注入到 userData） |
| `SC_HOST` / `SC_PORT` | `0.0.0.0` / `8000` | 监听地址/端口 |
| `SC_FRONTEND_DIST` | 空 | 指定后端托管的前端静态目录 |

## 4. 打包发布

统一入口：`scripts/package.py`（也可用根目录的 `package-*.ps1` / `package-*.sh` 包装脚本）。

```powershell
# 单独打某一种
python scripts/package.py --variant web      --platform windows
python scripts/package.py --variant desktop  --platform windows
python scripts/package.py --variant admin    --platform windows   # 授权签发端（内嵌私钥）
python scripts/package.py --variant tools    --platform windows   # 开发者工具（DB 工具 + 试用期工具）

# 一次性全打
python scripts/package.py --variant all --platform windows
```

> Debian 上把 `--platform` 换成 `debian`，并在 Debian 主机上执行。

### 产物目录

| variant | 输出目录 | 主要文件 |
| --- | --- | --- |
| web | `release/web/<platform>/` | 后端 exe + `frontend/` + `start-web.ps1` |
| desktop | `release/desktop/<platform>/` | NSIS 安装包 `wtcmd-platform-desktop-1.0.0-win-x64.exe` |
| admin | `release/license-admin/<platform>/` | `wtcmd-license-admin.exe`（**内嵌私钥，勿外发**） |
| tools | `release/dev-tools/<platform>/` | `wtcmd-db-tool.exe`、`wtcmd-expire-tool.exe`、`README.txt` |

> Windows 上打桌面版需保持 **Smart App Control（智能应用控制）关闭**，否则 NSIS 产物可能被拦截。

## 5. 配套工具用法详解

### 5.1 开发者工具（已打包为 exe，可给客户自助验证）

`release/dev-tools/windows/` 下两个 exe **已内嵌数据库主密钥**，无需任何额外密钥文件即可使用。客户端数据库 `data.db` 在磁盘上是整库加密的，普通工具打不开，用这两个工具即可解密查看 / 验证过期。

桌面版数据库默认位置（Windows）：

```
%APPDATA%\WTCMD Platform\data.db
```

#### (1) 数据库解密/加密/查看工具 `wtcmd-db-tool.exe`

```powershell
# 解密为明文 sqlite（可用 DBeaver / sqlite3 打开）
wtcmd-db-tool.exe decrypt --in "%APPDATA%\WTCMD Platform\data.db" --out plain.db

# 改完明文库后加密写回（覆盖客户端加密库）
wtcmd-db-tool.exe encrypt --in plain.db --out "%APPDATA%\WTCMD Platform\data.db"

# 直接查看授权状态 license_state（含防篡改校验）
wtcmd-db-tool.exe show --in "%APPDATA%\WTCMD Platform\data.db"
```

> 源码同款命令（开发机）：`python scripts/decrypt_db.py <decrypt|encrypt|show> ...`

#### (2) 试用期修改工具 `wtcmd-expire-tool.exe`

用于验证 “未激活客户端到期后是否被拦截”。它把 `license_state.first_seen` 改成 N 天前，并自动重算防篡改 `guard`。

```powershell
# 让试用期立刻过期（first_seen 设为 4 天前 > 3 天试用）
wtcmd-expire-tool.exe

# 指定还剩 1 天（first_seen 设为 2 天前）
wtcmd-expire-tool.exe --days-ago 2

# 清除已激活授权码，回到试用状态再测试
wtcmd-expire-tool.exe --clear-activation

# 指定数据库路径
wtcmd-expire-tool.exe --db "<data.db 路径>"
```

> 改动前请先 **关闭桌面客户端**，否则运行中的程序会再次落盘覆盖你的改动。
> 改完后重启桌面版（或重新调用 `/api/license/status`）即可看到试用期变化。

> 源码同款命令：`python scripts/expire_trial.py [--days-ago N] [--clear-activation] [--db ...]`

### 5.2 授权签发端 `wtcmd-license-admin.exe`

管理员给客户设备指纹签发授权码：

```powershell
# 命令行签发（也有 GUI）
release/license-tool/wtcmd-license-tool.py issue --fp <设备指纹> --days 365
```

> ⚠️ 该程序内嵌 **授权私钥**，仅限授权方内部使用，**绝不能随产品或源码分发给客户**。

### 5.3 密钥初始化脚本 `init_license_keys.py`

一次性生成 RSA 密钥对 + 数据库主密钥，并把它们以 “混淆 + 完整性校验” 形式编译进内嵌模块。详见[第 2.6 节](#26-生成密钥与内嵌模块关键一步)与[第 7 节](#7-密钥体系与备份重要)。

### 5.4 旧库升级验证脚本 `verify_legacy_upgrade.py`

见[第 6 节](#6-数据库加密与旧版本平滑升级)。

## 6. 数据库加密与旧版本平滑升级

### 6.1 加密模型

- 运行期：整个 SQLite 数据库驻留在 **内存连接** 中；磁盘上 `data.db` **始终是密文**。
- 文件格式：`MAGIC("WTSCDB01") || NONCE(12) || AES-256-GCM(密文+认证标签)`。
- 主密钥：编译进后端内部的内嵌混淆模块 `core._db_secret`，普通用户无法读取/替换。
- 每次提交：把内存库序列化 → 加密 → **原子写回**（tmp + fsync + `os.replace`）。

### 6.2 旧客户端如何无感知升级（数据安全保证）

老客户端的数据库是 **旧版明文 SQLite**（可能是 WAL 日志模式、且 schema 缺少新列）。当用户升级到当前加密版本后，后端启动逻辑（`database.py` 的 `_load_snapshot_from_disk` + `init_db`）会自动完成：

1. 识别磁盘文件是明文 SQLite，**整库读入内存**（WAL 模式会先 `wal_checkpoint(TRUNCATE)` + 切回 `journal_mode=DELETE`，避免序列化镜像无法装载）；
2. `CREATE TABLE IF NOT EXISTS` + 幂等 `ALTER TABLE` 自动补齐新表/新列（如 `projects`、`sessions`、`records.project_id/source/is_deleted` 等）；
3. 首次提交时把整库 **加密覆盖** 原文件（`WTSCDB01` 头）；
4. 清理 `-wal` / `-shm` 旁路文件。

结果：**旧数据零丢失、用户无感知**，磁盘自动从明文升级为密文。

### 6.3 自助验证

随时可运行验证脚本，它在临时目录里构造一个 “旧版明文 + WAL + 缺列” 的数据库，跑完整升级流程并逐项断言（绝不触碰真实数据库）：

```powershell
python scripts/verify_legacy_upgrade.py
```

预期最后输出：

```
==> 结论：旧版明文数据库可被当前版本无感知、零丢失地迁移为加密格式。VERIFIED OK
```

退出码 0 = 全部校验通过。脚本覆盖：初始为明文/WAL、升级后变密文、旁路文件清理、旧用户/旧记录完整保留、新列自动补齐、模拟重启后仍能从密文读回。

## 7. 密钥体系与备份（重要）

`scripts/keys/` 是 **整个授权与加密体系的根**，包含：

| 文件 | 作用 | 丢失后果 |
| --- | --- | --- |
| `private_key.pem` | RSA 私钥，签发授权码的唯一凭据 | 无法再签发能被现有客户端校验的授权码 |
| `public_key.pem` | RSA 公钥（编译进后端校验授权码） | 可由私钥重新导出 |
| `db_key.bin` | 32 字节数据库主密钥（AES-256-GCM） | **已加密的客户数据库将永久无法解密** |

由这三者派生的内嵌模块：`backend/core/_db_secret.py`、`backend/core/_license_pubkey.py`、`release/license-tool/wtcmd-license-tool.py`（均由 `init_license_keys.py` 生成）。

务必遵守：

- **离线、异地、加密备份** `scripts/keys/` 三个文件；它们不应进入可公开访问的仓库。
- 在新机器还原项目时，先还原这三个文件，再运行 `python scripts/init_license_keys.py`（不加 `--force`）刷新内嵌模块——这样历史授权码与历史加密数据库继续可用。
- 仅在确实要 **重置整个体系**（作废历史授权码、放弃旧加密库）时才用 `--force`。
- `dev-tools` 与 `license-admin` 内嵌了敏感密钥：`license-admin` 含私钥，**绝不外发**；`dev-tools` 含数据库主密钥，按内部工具管理，仅在需要让客户自助验证数据/过期时再分发。

## 8. 常见问题排查

| 现象 | 原因 | 处理 |
| --- | --- | --- |
| 后端启动报 `Embedded ... module not found` | 缺少内嵌密钥模块 | 运行 `python scripts/init_license_keys.py`（见 2.6） |
| 打包报 `Database master key not found` | 缺少 `scripts/keys/db_key.bin` | 还原备份或运行 `init_license_keys.py` |
| `wtcmd-db-tool` 报头部不匹配 | 传入的不是本系统加密库 | 确认 `--in` 指向真正的 `data.db` |
| 桌面版新建记录后看不到 | 多开了应用（旧问题，已加单实例锁） | 用新版安装包；勿手动多开 |
| PowerShell 把打包 INFO 日志当错误、误报失败 | 原生命令 stderr 被 PS 视为错误 | 用 `Start-Process ... -RedirectStandardError err.log; $p.ExitCode` 判断真正退出码 |
| Windows 打桌面版被拦截/失败 | Smart App Control 开启 | 关闭智能应用控制后重试 |
| 控制台中文/emoji 乱码 | GBK 控制台 | 脚本输出使用 ASCII 标记即可 |
